035/**
036 * 
037 * The scenario is the entry point to MATSim
038 * scenarios. An implementation of Scenario
039 * has to provide implementations
040 * for the different return types, e.g. Network,
041 * Facilities or Population.
042 * 
043 * <h3> Design aspects </h3>
044 * <p>
045 * Scenario (and in particular {@link MutableScenario}) now accepts arbitrary implementations of the contributing containers.  As stated
046 * below, in some sense the whole scenario container is no longer necessary with the injection framework.  The containers themselves
047 * should allow replaceable implementations, see {@link MatsimToplevelContainer}.
048 * </p>
049 * <h3> History </h3>
050 * <p>
051 * Originally, MATSim provided "global" variables such as access to the network with public static variables. 
052 * </p><p>
053 * This was changed around 2010 towards an approach where everything was moved around by non-static references.  The Scenario 
054 * container was thus introduces to contain most of the objects that were public static before, so that reaching around one reference
055 * would be enough for most purposes.  Reasons for that change include(d):<ul>
056 * <li> Some people wanted to be able to have, say, two networks and/or two populations simultaneously in the code.  For example in order to compare them.
057 * This was quite difficult with code that used public static variables.  (Clearly, one could put the desired scenario or network to the public
058 * static reference, run the method, and then change it back.  Yet, ...)
059 * <li> We never noticed it a lot since consistent regression testing was introduced <i> after </i> this change.  Yet when I introduce regression
060 * tests for other software that uses public static variables excessively, I have to use a mode that constructs a new JVM every time a test
061 * is started.  This works for integration tests since they use a fair amount of time anyways, but presumably makes unit testing much slower. 
062 * <li> As an argument going into the same direction: Without global static variables, a class or method has access to what is given into 
063 * it, but not more.  With global static variables, any class or method has access to everything that can be accessed that way.  This means
064 * that one can never rely on a class or method to not change more than is visible from the outside, making reliably encapsulated modules 
065 * close to impossible.
066 * </ul>
067 * </p><p>
068 * Around 2015, this was superseded by the (guice) injection approach.  Since with that it is possible to get the elements of the scenario
069 * injected separately, the scenario container essentially does not seem to be necessary any more.  In retrospect, it might have been easier
070 * to move from the public static approach to the injection approach right away, but we did not know this at that time.   
071 * </p>
072 * 
073 * @author dgrether
074 * @author (of documentation) Kai Nagel
075 */
076public interface Scenario {
077
078        Network getNetwork();
079
080        Population getPopulation();
081
082        TransitSchedule getTransitSchedule();
083        
084        Config getConfig();
085
086        /**
087         * Adds the given object to the scenario, such it can be
088         * retrieved with {@link #getScenarioElement(String)} using
089         * the name given here as a key.
090         * <br><br>
091         * Design issues:<ul>
092         * <li> This is currently set to deprecated, since the guice injection approach achieves something similar.  However, 
093         * at this point injection is only possible after {@link Controler} was started, which stands in the way of <i>first</i> constructing
094         * the scenario and only <i>then</i> starting {@link Controler}.  So maybe we should un-deprecate this method.  Kai/Theresa, sep'16
095         * Un-deprecated for the time being.   Kai, dec'16
096         * </ul>
097         *
098         * @param name the name to which the object should be associated
099         * @param o the object. <code>null</code> is not allowed.
100         * 
101         * @throws NullPointerException if the object is null
102         * @throws IllegalStateException if there is already an object
103         * associated to this name.
104         */
105//      @Deprecated // see javadoc
106        void addScenarioElement(String name, Object o);
107
108        /**
109         * See the "Design issues" under {@link Scenario#addScenarioElement(String, Object)}.
110         *
111         * @param name the name of the element to get
112         * @return the object associated with that name, or null if none is associated
113         */
114//      @Deprecated // see javadoc
115        Object getScenarioElement(String name);
116
117        ActivityFacilities getActivityFacilities();
118
119        Vehicles getTransitVehicles();
120
121        Vehicles getVehicles();
122
123        Households getHouseholds();
124
125        Lanes getLanes();
126
127}
