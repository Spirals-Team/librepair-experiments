027/**
028 * Design remark(s):<ul>
029 * <li> These need to be factories since, at least for threads, we may need multiple
030 * instances.  (If nothing else, each may set its own person.) kai, mar'12
031 * <li> In principle, the above statement is probably no longer correct, since the person is (now) contained 
032 * in the getLinkTravelDisutility(...) call.  However, there is a danger that someone memorizes the person in a class-global 
033 * (= private) field ... and then the class is no longer thread safe.  Thus, we may be safer by having one instance of
034 * of TravelDisutility per router thread.  kai/michaelz/dgrether, apr'13
035 * <li> Seems to me that the above arguments are obsolete with guice ... each inject could pull a new instance.  HOWEVER, it seems that we 
036 * either need to pass the TravelTime object into the creator, or at least the mode (so it can pull the correct TravelTime).  Neither is
037 * possible with a syntax of type
038 * <pre>
039 *    addTravelDisutilityBinding(mode).to???</pre>
040 * The only option I can think about would be something like
041 * <pre>
042 * addTravelDistuilityBinding(mode).toProvider( new TravelDisutilityProvider(mode) ) ;
043 * </pre>
044 * However, MZ says that there are also instances that pass non-standard TravelTime objects into the standard TravelDisutiity, and 
045 * such a behavior would be difficult to express when extension point would be the TravelDisutility, and not its factory.  kai, feb'17
046 * </ul>
047 * 
048 * @author dgrether
049 *
050 */

052        
053        public TravelDisutility createTravelDisutility(TravelTime timeCalculator);
054
055}
